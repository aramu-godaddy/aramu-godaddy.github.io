Settings & Environment Variables

5 items
Settings

Default to Haiku

Every session starts on the cheapest model (~18x cheaper than Opus). Escalate only when the task demands it.

// ~/.claude/settings.json
{ "model": "haiku" }
Settings

Medium Effort Level

Reduces output verbosity and reasoning depth, directly cutting output tokens per response.

"env": {
  "CLAUDE_CODE_EFFORT_LEVEL": "medium"
}
Settings

Disable Non-Essential Model Calls

Prevents background and speculative model invocations that burn tokens without direct user benefit.

"env": {
  "DISABLE_NON_ESSENTIAL_MODEL_CALLS": "1"
}
Settings

Disable Experimental Betas

Avoids beta features that may trigger additional model calls behind the scenes.

"env": {
  "CLAUDE_CODE_DISABLE_EXPERIMENTAL_BETAS": "1"
}
Settings

Autocompact at 75%

Trigger context compression earlier. Late compression is aggressive and lossy — Claude re-reads files it already saw, paying for the same content twice.

"env": {
  "CLAUDE_AUTOCOMPACT_PCT_OVERRIDE": "75"
}

Why This Matters

Default (near 100%)At 75%
When it triggersContext nearly full25% headroom remaining
Compression qualityAggressive, lossyGentle, preserves key details
File re-reads neededYes — lost detailsNo — details retained
Net effectPays for same content twiceReads once, keeps working

Earlier compression is incremental and preserves more detail, avoiding duplicate file reads that double your input token cost.

Workflow

Minimize Tool Output

Every line of tool output enters the context window and gets re-sent as input tokens on every subsequent turn. The same information can cost wildly different amounts depending on how you ask for it.

## Conventions
- Minimize tool output:
  - git log -5 --oneline
  - git diff --stat before full diffs
  - Read with offset/limit
  - Grep with head_limit and glob filters
  - Run specific test files, not full suites

Per-Tool Comparison

Tool CallUnconstrainedWith RulesReduction
git log~50 lines~5 lines10x
git diff~200 lines~15 lines (--stat first)13x
Read file~500 lines (full file)~30 lines (offset/limit)17x
Grep search~100 lines~10 lines (head_limit + glob)10x
Test runFull suite (~300 lines)Specific files (~20 lines)15x
Per cycle~850 lines~60 lines~14x less

Over a 20-turn session with 3 investigation cycles, this saves ~47,000 lines of re-sent context.

Workflow

Plan Before Implementing

Require Claude to present a plan and get approval before implementing. Without this, Claude may edit files, hit errors, and backtrack — every wrong-path action costs tokens that produce zero value.

## Workflow
- Present a plan and get approval
  before implementing.
  No autonomous multi-file changes
  or test edits.

Cost Comparison

No PlanPlan First
First actionStarts editing immediatelyPresents plan (~200 tokens)
Wrong approachUndo + redo all edits (2x)Redirect before code written
Typical waste5,000+ tokens thrown away~0 (caught at plan stage)
Net effectPays full cost for wrong path~200 tokens to validate

A wrong-path implementation burns 5,000+ output tokens on edits, test runs, and error loops — all thrown away.

Workflow

Subagents for Exploration

Everything Claude reads goes into the main context window and gets re-sent every turn. A subagent runs in its own isolated context — only a concise summary comes back.

## Workflow
- Use the investigator subagent
  for multi-file exploration
  before making changes.

The Math

Without SubagentWith Subagent
Where data lands3,000 lines in main context3,000 lines in subagent (isolated)
Main context cost3,000 lines × 20 turns = 60k30 lines × 20 turns = 600
Subagent cost—3,000 lines × 1 turn = 3k
Total~60,000 lines~3,600 lines (17x less)

The longer the session runs after exploration, the bigger the savings compound.

Search Before Creating

Before adding constants or data structures, search for existing definitions. Avoids duplication that a later review cycle would catch — saving a full fix round-trip.

  structures, search the package for
  existing definitions to avoid
  duplication.

Catches duplication and waste in one pass rather than multiple correction rounds. Run it before committing while context is fresh.

Memory Rules

1 item
Memory

Lean Commit / Push / PR Workflow

When Claude already made the changes in the session, pre-commit reads are pure waste. This memory rule eliminates redundant calls:

  1. Skip reads entirely — even git diff --stat + git status + git log -5 --oneline is 3 unnecessary calls when Claude already knows what changed.
  2. Chain into 1 call instead of 3 separate ones (add, commit, push).

Goes from ~4–6 tool calls down to 1. The CLAUDE.md rules handle output size — this memory handles redundant calls.

# ~/.claude/memory/feedback_commit_push_workflow.md
# When user asks to commit/push/PR after a session where
# you made the changes:

- Skip git status, git diff, git log — you already know
- Single Bash call:
  git add <files> && git commit -q -m "..." && git push -q
- PR:
  gh pr view --json url 2>/dev/null || gh pr create ...

Hooks

2 items
Hook

Pre-Commit Lint

A PreToolUse hook runs ESLint on staged files before git commit. Without it, each failed commit triggers a costly retry loop.

// settings.json
"hooks": {
  "PreToolUse": [{
    "matcher": "Bash",
    "hooks": [{
      "type": "command",
      "command": "bash ~/.claude/hooks/lint-staged.sh",
      "timeout": 30
    }]
  }]
}

Retry Cost Breakdown

No Lint HookWith PreToolUse Hook
When errors foundAfter commit failsBefore commit runs
Per error cost3 calls (read error + edit + retry)1 call (fix + commit)
Multiple errorsCascades: 6–9 wasted callsAll caught in single pass
Net effectRetry loop per lint errorFix once, commit once

The hook intercepts git commit calls and runs ESLint on staged files. All issues surface in one pass before any commit attempt.

Hook

Cost-Aware Status Line

Live dollar-cost display in your prompt. See spend in real time and intervene early if a session gets expensive. Color-coded: green < $1, yellow $1–$5, red > $5.

// settings.json
"statusLine": {
  "type": "command",
  "command": "bash ~/.claude/statusline-command.sh"
}
